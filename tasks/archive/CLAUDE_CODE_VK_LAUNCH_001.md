## 3. Шаг 0. Подготовка и pre-flight checks

### 3.1. Чтение контекста (5 минут)

3. Посмотреть `routes/api.py` — endpoints `/api/vk/status`, `/api/posts/<id>/publish-now`, `/api/posts/<id>/schedule`
4. Создать ветку `git checkout -b vk-launch/v0.2.1-validation`

### 3.2. Health-check приложения
1. `python app.py` — стартует без ошибок
2. `curl -s http://127.0.0.1:5000/api/vk/status` — должен вернуть `{"configured": true}` (если `false` — стоп, проверить `.env`, не идти дальше)
3. `curl -s http://127.0.0.1:5000/api/rubrics | jq length` — должен вернуть `8`
4. Проверить логи: `tail -n 30 logs/app.log` — никаких ERROR/CRITICAL при старте

### 3.3. Прямая проверка токена и group_id (без участия нашего кода)
Перед интеграционным тестом убедиться, что токен живой и `group_id` корректный — это сэкономит часы диагностики если что-то не так с конфигом.

Запустить из Python (одноразовый скрипт `scripts/vk_smoke.py`, после теста удалить):
```python
import os, httpx
from dotenv import load_dotenv
load_dotenv()
token = os.environ["VK_COMMUNITY_TOKEN"]
group_id = os.environ["VK_GROUP_ID"]
v = os.environ.get("VK_API_VERSION", "5.199")
r = httpx.get(
    "https://api.vk.com/method/groups.getById",
    params={"group_id": group_id, "access_token": token, "v": v},
    timeout=15
).json()
print(r)
```

Ожидаемый ответ — JSON с `response: [{ id: <int>, name: "Лора | Английский с AI-репетитором", screen_name: "lora_ai_english", ... }]`. Если в ответе `error` с кодом 5 (token invalid), 15 (access denied), 100 (one of params invalid) — **остановиться**, описать ошибку пользователю, не идти дальше тестов. Имя сообщества и `screen_name` сверить с ожидаемым; если не совпадает — `group_id` в `.env` указывает на другое сообщество.

После проверки удалить скрипт `scripts/vk_smoke.py`.

## 4. Шесть сценариев тестирования

Все сценарии прогонять последовательно. После каждого УСПЕШНОГО — попросить пользователя удалить пост в VK вручную (через меню «...» на посте), прежде чем переходить к следующему. После каждого тестового поста — фиксировать в отчёте: время, vk_post_url, статус в БД, успех/ошибка.

Если сценарий **падает** — НЕ продолжать дальше «авось получится со следующим». Остановиться, диагностировать, принять решение по фиксу (раздел 5).

### Сценарий A — Только текст (без медиа)

**Сложность:** простейший случай, проверяет базовый `wall.post` без attachments.

1. Через UI создать пост:
   - Рубрика: «Свободная тема ✨»
   - Описание задачи: `Напиши тестовый пост из одного предложения для проверки публикации: "Тест A — публикация только текста, скоро будет удалён."`
   - Сгенерировать текст (Claude Sonnet)
   - Сохранить как черновик
2. В библиотеке найти этот черновик → нажать «📤 Опубликовать сейчас» → подтвердить.
3. **Ожидание:** toast «Опубликовано», открывается новая вкладка с постом в VK, статус черновика → `published`, поля `vk_post_id` и `vk_post_url` заполнены, `published_at` установлен.
4. Записать `vk_post_url` в отчёт.
5. Попросить пользователя удалить пост в VK.

### Сценарий B — Текст + сгенерированная картинка (Kling)

**Сложность:** проверяет `photos.getWallUploadServer` + `photos.saveWallPhoto` + `wall.post` с attachment.
**Условие пропуска:** если у пользователя баланс Kling исчерпан — пропустить, отметить в отчёте.

1. Создать пост:
   - Рубрика: «Слово дня» с темой `serendipity`
   - Сгенерировать текст
   - Сгенерировать картинку через Kling (1–2 минуты ожидания, проверить что таймер работает)
   - Сохранить как черновик
2. Опубликовать сейчас.
3. **Ожидание:** пост в VK с прикреплённой картинкой, картинка отображается корректно (не битая, размер сохранён).
4. Записать в отчёт.

### Сценарий C — Текст + картинка из медиа-библиотеки

**Сложность:** проверяет тот же VK-пайплайн но с пред-загруженным изображением.

1. Загрузить через вкладку «Медиа» любой test image (можно сгенерировать через Pillow `Image.new('RGB', (800,800), '#C8862C').save('test_assets/test_image.png')` если нечего загрузить).
2. Создать пост: рубрика «Минифакт», тема `English alphabet has 26 letters`, сгенерировать текст.
3. Через «Выбрать из библиотеки» прикрепить только что загруженную картинку.
4. Сохранить → Опубликовать сейчас.
5. **Ожидание:** пост в VK с прикреплённой картинкой, ровно той что выбрали.

### Сценарий D — Текст + видео

**Сложность:** проверяет `video.save` + загрузку файла. ВК обрабатывает видео асинхронно — после публикации пост появляется, видео может быть «обрабатывается» несколько минут. Это нормально, не баг.

1. Создать пост: рубрика «Свободная тема», описание задачи `Короткий пост-приветствие на 2 предложения с видео-демонстрацией`.
2. Сгенерировать текст.
3. Загрузить `test_assets/test_video.mp4` через медиа-вкладку → прикрепить к посту.
4. Сохранить → Опубликовать сейчас.
5. **Ожидание:** пост в VK с прикреплённым видео. Видео может несколько минут показывать «обрабатывается» — это норма.
6. **Особое внимание к логам:** в `ai_calls.jsonl` должны быть две успешные записи: `vk:upload_video` и `vk:wall_post`. В `app.log` не должно быть warnings про время загрузки.

### Сценарий E — Запланированная публикация на +6 минут

**Сложность:** проверяет APScheduler, фоновое задание, корректное время.

1. Создать пост: рубрика «Типичная ошибка», тема `make a photo vs take a photo`.
2. Сгенерировать текст. Без картинки.
3. Сохранить → нажать «🕐 Запланировать» → выбрать время **текущее +6 минут** в UTC локали (datetime-local picker автоматически в локальной зоне; убедиться что не +6 секунд из-за миса с timezone).
4. Подтвердить.
5. **Ожидание сразу:** статус черновика → `scheduled`, в библиотеке появляется горчичная плашка «Через 6 минут» (или сколько осталось), кнопка «Отменить расписание» доступна.
6. Засечь время. Подождать 7 минут (с запасом).
7. **Ожидание после 7 минут:** статус → `published`, `vk_post_url` заполнен, пост виден в VK.
8. **Не закрывать `app.py` во время ожидания** — это базовый тест scheduler-а в прямом режиме.

### Сценарий F — Запланированная публикация ВЫЖИВАЕТ перезапуск

**Сложность:** проверяет восстановление задач при старте + догон пропущенных.

1. Создать пост: рубрика «Деловой английский», тема `polite refusal of a meeting`.
2. Сгенерировать текст. Без картинки.
3. Запланировать на текущее время **+12 минут**.
4. Убедиться что задача в БД (`SELECT * FROM apscheduler_jobs;` — должна быть 1 запись для этого post_id).
5. **Ctrl+C** — корректно остановить `python app.py` (проверить что закрылся без ошибок, не оставил процессов).
6. Подождать 1 минуту. Снова `python app.py`.
7. В логах при старте должно быть сообщение типа «restored N scheduled jobs» или подобное.
8. Дождаться запланированного времени + 1 минута.
9. **Ожидание:** пост опубликован, статус `published`.

**Опционально (если время позволяет) — F2: тест догона пропущенных:**
1. Создать ещё один пост, запланировать на +5 минут.
2. **Ctrl+C** через 30 секунд.
3. Подождать 7 минут (запланированное время УЖЕ прошло).
4. Запустить `python app.py`.
5. **Ожидание:** при старте scheduler видит просроченную задачу и публикует её немедленно. В логах сообщение про «catch-up» или «overdue». Статус → `published`.

## 5. Что делать если сценарий упал

### 5.1. Если ошибка очевидная и фиксится в 5–10 строках
Например: опечатка в названии метода VK, неверный формат datetime, забыли передать параметр.
- Зафиксировать корневую причину одной строкой в отчёте
- Сделать минимальный фикс
- Закоммитить отдельно: `fix(vk): краткое описание (vk-launch-001)`
- Перезапустить упавший сценарий
- Если после фикса прошёл — продолжать дальше с того же места

### 5.2. Если ошибка требует серьёзной переделки
Например: неверная архитектура клиента, race condition в scheduler, проблема с миграцией БД.
- **Не пытаться чинить здесь и сейчас.** Это уже не валидация v0.2.1, это hotfix-002.
- Зафиксировать в `docs/sessions/2026-05-XX-vk-launch.md`:
  - Что не работает
  - Гипотеза причины
  - Какой scope фикса нужен
- Если сценарий, который упал, был A или B (базовый текст или картинка) — остановить тестирование, базовая публикация не работает.
- Если упал C/D/E/F — отметить, попробовать продолжить остальные (они независимы), но в финальном отчёте чётко обозначить «v0.2.1 НЕ готов до фикса X».

### 5.3. Типичные ошибки VK API и что они значат

| Код | Сообщение | Что значит | Что делать |
|---|---|---|---|
| 5 | User authorization failed | токен невалидный/истёк | пользователь генерирует новый ключ, кладёт в .env |
| 7 | Permission denied | у токена нет нужного scope | пересоздать ключ со всеми галочками wall, photos, video, stories |
| 9 | Flood control | слишком много запросов | подождать, уменьшить частоту тестов |
| 14 | Captcha required | VK требует капчу | требует ручного решения, обычно перезапуск с задержкой решает |
| 15 | Access denied (group) | неверный group_id, или токен не от этой группы | проверить `groups.getById` (см. шаг 3.3) |
| 100 | Invalid params | в attachments опечатка/невалидный формат | проверить формат `photo{owner_id}_{id}` |
| 214 | Access to adding post denied | сообщество запрещает посты по политике | проверить настройки сообщества → Управление → Сообщения |
