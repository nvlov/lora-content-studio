# CLAUDE_CODE_HOTFIX_001 — Модалка медиа-библиотеки зависает при загрузке

> **Тип:** hotfix к v0.2.0 (не новая итерация)
> **Целевая версия после фикса:** v0.2.1
> **Приоритет:** блокирующий — приложение неработоспособно

---

## 1. Симптом (что видит пользователь)

При открытии `http://127.0.0.1:5000` (вкладка «Создать пост» по умолчанию):
- **Сразу при загрузке** всплывает модальное окно «Выбор медиа из библиотеки», хотя пользователь ничего не нажимал
- Внутри модалки **только заголовок и кнопка «Закрыть»** — никакого списка медиа, никакого спиннера, никакого сообщения «пусто»
- Кнопка **«Закрыть» не реагирует** на клики
- Фон страницы затемнён, **весь интерфейс заблокирован** — невозможно создать пост, перейти на другие вкладки, нажать кнопки навигации
- `Esc` тоже не закрывает (хотя в v0.2 заявлено что Esc закрывает все модалки)

Воспроизведение: 100% при каждой загрузке страницы. Состояние БД: один пост, мигрированный с v0.1, один MediaAsset (image), `.env` с прописанными VK_COMMUNITY_TOKEN и VK_GROUP_ID.

## 2. Артефакты от пользователя (приложены к этому файлу)

## 3. Гипотезы причин (по приоритету)

### H1 — JS-ошибка при инициализации блокирует event handlers (вероятность ~50%)
Где-то в `static/js/app.js` при `DOMContentLoaded` падает что-то типа `Cannot read properties of null` или `is not a function`. После падения handlers не навешиваются, поэтому:
- Модалка случайно показана через CSS-класс установленный в HTML или сразу в JS
- Click на «Закрыть» не отрабатывает потому что слушатель не успел навеситься
- Esc тоже не работает по той же причине

**Признак:** красная строка в Console DevTools на этапе загрузки страницы.

### H2 — Модалка открывается ошибочно при загрузке (вероятность ~25%)
Где-то в коде вызов вида `openMediaPicker()` или `showModal('media-picker')` стоит вне обработчика клика — например, в инициализации `state` или при восстановлении черновика. Открывается, потом её содержимое (`fetch /api/media`) падает в `.catch`, который не закрывает модалку.

**Признак:** в Network видно неудачный запрос к `/api/media` (HTTP 500 / 404 / CORS), при этом в Console может не быть JS-ошибки.

### H3 — Бэкенд `/api/media` валится из-за миграции v0.1 → v0.2 (вероятность ~15%)
Таблица `media_assets` создалась, но какое-то поле/индекс не на месте. Запрос валится с 500. `app.log` покажет stacktrace.

**Признак:** красные строки в `logs/app.log` со словами `OperationalError`, `no such column`, `IntegrityError`.

### H4 — Кнопка «Закрыть» имеет неверный селектор/обработчик (вероятность ~10%)
Опечатка в коде: например, обработчик навешивается на `#media-picker-close` а в HTML кнопка имеет id `media-picker__close`. Сама модалка нормально открыта, но закрыть её невозможно.

**Признак:** в Console при клике на «Закрыть» нет реакции вообще, ошибок тоже нет; в Elements DevTools видно что кнопка существует, но Event Listeners на ней нет.

## 4. План диагностики и фикса

### Шаг 0. Подготовка
1. Прочитай отчёт сессии 2: `docs/sessions/2026-05-10-session-2.md` — что именно реализовано в Блоке C (медиа-студия) и в подразделе «Видео в редакторе».
2. Посмотри `static/js/app.js` — особенно блоки media-picker, media-library, и общая инициализация на DOMContentLoaded.
3. Посмотри `templates/index.html` — разметку модалки media-picker, начальные классы.
4. `git checkout -b hotfix/v0.2.1-media-picker`

### Шаг 1. Воспроизвести локально
1. Запустить `python app.py` чисто (с теми же `.env` и БД, что у пользователя).
2. Открыть в браузере, повторить сценарий. **Если воспроизводится у тебя** — переходи к Шагу 2.
3. **Если НЕ воспроизводится** — проблема в данных пользователя или окружении. Сделать так:
   - Запросить у пользователя дамп `data/lora.db` (или хотя бы `SELECT * FROM media_assets; SELECT * FROM posts WHERE deleted_at IS NULL;` через `sqlite3 data/lora.db`)
   - Запросить версию браузера и Windows
   - Запросить точный `logs/app.log` (последние 200 строк) и `logs/ai_calls.jsonl` (последние 50 строк)
   - Не двигаться дальше пока проблема не воспроизведена

### Шаг 2. Найти корневую причину
1. По скриншоту Console (или своему воспроизведению) определить какая из гипотез H1–H4 подтверждается.
2. **Не накладывать фикс наугад.** Если корневая причина не очевидна — добавить временный `console.log()`/`logger.debug()` в подозрительные места, перезапустить, наблюдать.
3. Зафиксировать корневую причину одной фразой в отчёте.

### Шаг 3. Минимально-инвазивный фикс

**Принципы:**
- Меняем **только то что сломано**. Не рефакторим попутно. Не «улучшаем» соседние блоки.
- Не правим CSS, JS, Python, HTML «заодно». Каждое изменение должно быть оправдано тем что без него баг не уходит.
- Если по дороге нашёл другие подозрительные места — записать в отчёт как «потенциальные риски, на следующую итерацию», но **не трогать**.

**Возможные конкретные фиксы по гипотезам:**

- **Под H1:** найти строку которая падает, окружить try/catch с понятным `console.error` И/ИЛИ исправить причину падения (часто — обращение к элементу до его появления в DOM, лечится переносом инициализации внутрь `DOMContentLoaded` или добавлением guard `if (!el) return;`).

- **Под H2:** убрать ошибочный вызов открытия модалки при загрузке. В `.catch` блока `fetch('/api/media')` обязательно закрывать модалку и показывать toast с ошибкой.

- **Под H3:** исправить миграцию или добавить недостающий ALTER TABLE. Идемпотентно. Проверить `PRAGMA table_info(media_assets)` до и после.

- **Под H4:** исправить селектор / id, убедиться что обработчик навешен.

**После фикса в любой гипотезе:** убедиться что `Esc` тоже закрывает модалку (это отдельный обработчик — он тоже мог пострадать).

### Шаг 4. Регрессионный smoke-test после фикса

Прогнать без VK API (если у пользователя VK не настроен):
1. Главная страница загружается чисто, модалок нет
2. Выбрать рубрику «Слово дня» → ввести тему «procrastinate» → сгенерировать текст → текст появился
3. Сохранить как черновик → перейти в «Библиотека» → черновик виден
4. Открыть черновик → текст загрузился в редактор
5. Перейти на «Медиа» → drag-n-drop тестового image (любая картинка) → загрузилась, появилась в сетке
6. Вернуться в «Создать пост» → нажать «Выбрать из библиотеки» (та самая модалка) → **открылась штатно с содержимым**, кнопка «Закрыть» работает, Esc работает
7. Выбрать загруженную картинку → прикрепилась к посту, превью VK обновилось
8. Перейти в «Настройки» → одна из рубрик открывается, поля редактируются
9. `Ctrl+S` — сохраняет, `Ctrl+Enter` — генерирует, `Esc` — закрывает любую модалку
10. Удалить тестовый черновик → модалка подтверждения → удалился (soft-delete)

Если у пользователя VK настроен — также пункты 11-12:
11. Кнопки «Опубликовать сейчас» и «Запланировать» активны (не серые)
12. `GET /api/vk/status` возвращает `{configured: true}` (можно проверить через DevTools или curl)

**Критично:** все 10–12 пунктов должны пройти. Если хоть один сломан — это новый баг, фиксить отдельно (или откатывать hotfix и думать заново).

## 6. Если фикс невозможен из-за нехватки информации

Если после двух-трёх попыток воспроизвести и собрать данные с пользователем баг остаётся неуловимым — **не накладывать «авось поможет» фикс**. Вместо этого:
1. Добавить временный verbose-logging в подозрительные места
2. Дать пользователю инструкцию: «запусти, открой DevTools, повтори сценарий, пришли вот эти конкретные строки из Console и app.log»
3. Дождаться данных, тогда фиксить

Лучше потратить 20 минут на правильную диагностику, чем 2 часа на починку через изменение случайных файлов и потом два дня на регресс.
