## 4. Блок A — UX-полировка и безопасность

### A1. Иконочные кнопки

В редакторе поста и в карточках библиотеки заменить текстовые кнопки на иконочные с подписью-тултипом:

| Действие | Иконка (SVG inline или unicode) | Где появляется |
|---|---|---|
| Редактировать | ✏️ или SVG карандаш | карточка библиотеки |
| Сохранить | 💾 или SVG дискета | редактор (внизу), карточка библиотеки |
| Опубликовать сейчас | 📤 или SVG paper plane | редактор, карточка |
| Запланировать | 🕐 или SVG часы | редактор, карточка |
| Удалить | 🗑️ или SVG корзина | карточка библиотеки |
| Дублировать («Создать на основе») | 📋 или SVG copy | карточка библиотеки |

Используй inline SVG из открытых наборов (Heroicons / Lucide) — копируй пути напрямую в HTML, не подключай иконочные библиотеки. Размер иконки 20px, цвет — `var(--color-primary)`. Все иконки имеют `aria-label` и `title` (тултип) на русском.

Кнопки с деструктивным действием (удалить) — отдельный класс с цветом `var(--color-error)` при hover.

### A2. Подтверждения деструктивных и неотменимых действий

Реализовать переиспользуемый модальный диалог `confirm(title, message, confirmLabel, cancelLabel, type)` в `app.js`, возвращающий Promise<boolean>.

Применить на:
- **Удаление черновика** — «Удалить черновик? Это действие нельзя отменить.»
- **Публикация в VK сейчас** — «Опубликовать пост в сообществе ВК прямо сейчас? Удалить или отредактировать после публикации можно только вручную в самом ВК.»
- **Запланированная публикация** — «Запланировать публикацию на {datetime}? Пост будет опубликован автоматически.»
- **Уход с несохранёнными изменениями** — `beforeunload` event если в редакторе есть несохранённые правки.

Стиль модалки — карточка по центру, затемнённый фон (`rgba(31,58,46,0.4)`), две кнопки. Esc закрывает с false, Enter подтверждает с true.

### A3. Soft-delete черновиков

Добавить в модель `Post` поле `deleted_at` (datetime, nullable). При удалении не удалять запись физически, а ставить `deleted_at = now()`. Все запросы `GET /api/posts` фильтруют `deleted_at IS NULL`.

Добавить эндпоинт `POST /api/posts/<id>/restore` (восстановление). В UI пока не делаем кнопку восстановления — она появится в v0.3. Достаточно того, что данные не теряются.

### A4. Прогресс-индикаторы для длительных операций

Любая операция длительностью >2 секунд должна:
1. Блокировать кнопку, которая её запустила (`disabled` + opacity 0.6)
2. Показывать спиннер-кружок рядом с кнопкой
3. Если операция длится >10 секунд — добавить таймер «Идёт {N} сек…»
4. Показывать toast при завершении (success/error)

Применить на:
- Генерация текста (Claude, обычно 3–8 секунд)
- Генерация картинки через Kling (1–2 минуты — ОБЯЗАТЕЛЬНО таймер)
- Генерация промптов для медиа (Claude, 3–8 секунд)
- Публикация в VK (5–15 секунд для текста, до минуты для видео)
- Загрузка файла (зависит от размера)

Для Kling специально: показать строку «Это занимает 1–2 минуты, не закрывай вкладку».

### A5. Лимит символов VK + индикация

Лимит текста поста в ВК — 4096 символов с пробелами. Под полем редактора текста — счётчик `{N} / 4096`. Цвет:
- `<3500` — нейтральный (text-muted)
- `3500–4000` — предупреждение (color-accent / горчичный)
- `>4000` — ошибка (color-error / красный) и блокировка кнопок «Опубликовать сейчас» / «Запланировать»

### A6. Toast-система

Заменить разрозненные `alert()` и inline-сообщения на единую систему тостов:
- Контейнер `<div id="toast-container">` в правом нижнем углу
- Функция `toast(message, type, duration)` где type ∈ `success | warn | error | info`, duration по умолчанию 5000мс
- Очередь тостов (один над другим, плавное появление-исчезание через CSS-анимацию)
- На клик — закрытие
- Иконка слева в зависимости от типа

### A7. Hot-keys

Добавить горячие клавиши в редакторе:
- `Ctrl+Enter` — генерировать текст / регенерировать
- `Ctrl+S` — сохранить как черновик
- `Esc` — закрыть модалку
- `Ctrl+P` — опубликовать сейчас (с подтверждением)

В UI — мини-подсказка под кнопкой: `Ctrl+S` мелкими буквами рядом с иконкой сохранения.

#### C1.1. Генератор промптов для image/video

Workflow по согласованной схеме (см. ответ #3 в чате обсуждения):

1. Поле «Опишите идею на русском» (textarea, placeholder: «Лора объясняет грамматику у доски в уютном классе»)
2. Радио-переключатель: **Промпт для изображения** | **Промпт для видео** (по умолчанию — изображение)
3. Дропдаун «Стиль» с пресетами (можно расширять):
   - `pixar_3d_brand` — фирменный стиль (Pixar-3D, кремовый+горчичный+тёмно-зелёный) — **по умолчанию**
   - `photo_realistic` — фотореалистичный

   - `cinematic` — кинематографический (для видео особенно)
4. Дропдаун «Соотношение сторон»: `1:1`, `9:16` (vertical / Stories), `16:9`
5. Кнопка «Сгенерировать промпты» → Claude возвращает **3 варианта** детального английского промпта
6. Каждый вариант отображается в карточке с:
   - Кнопкой «📋 Копировать»
   - Кнопкой «💾 Сохранить» (для повторного использования — таблица `MediaPrompt`)
   - Подписью «для какого сервиса оптимизирован»: Kling AI / utopy.ai / MidJourney (общая рекомендация)

### C3. Очистка таймауту Kling в реальной нагрузке

В v0.1 `KlingImageProvider` имел timeout 60 сек на генерацию. По опыту — Kling иногда отдаёт результат за 80–120 секунд. Увеличить timeout до **180 секунд**, опрос статуса каждые 5 сек. При истечении — показать пользователю понятное сообщение «Kling не успел за 3 минуты. Попробуй ещё раз или используй ручную загрузку».

### D1. Подготовка VK-приложения и токена (раздел README)

В README добавить раздел «Настройка VK API» с пошаговой инструкцией для пользователя:

   (Подставить APP_ID вместо `{APP_ID}`. Обязательно авторизоваться под админом сообщества `lora_ai_english`.)

   ⚠ **Важно:** для публикации от имени сообщества нужен именно **community access token**, не user token. Получается через:
   - Личный кабинет сообщества → Управление → Работа с API → Создать ключ

### D2. Клиент VK API (`core/vk_client.py`)

Реализовать класс `VKClient` с методами:

```python
class VKClient:
    def __init__(self, token: str, group_id: int, api_version: str = "5.199"):
        ...

    def upload_photo(self, image_path: str) -> dict:
        """
        Загружает фото на стену сообщества.
        1) photos.getWallUploadServer(group_id) → upload_url
        2) POST файл на upload_url → server, photo, hash
        3) photos.saveWallPhoto(group_id, photo, server, hash) → owner_id, id
        Возвращает: {"attachment": "photo{owner_id}_{id}"}
        """

    def upload_video(self, video_path: str, name: str, description: str = "") -> dict:
        """
        Загружает видео в сообщество.
        1) video.save(group_id, name, description, wallpost=0) → upload_url, owner_id, video_id
        2) POST файл на upload_url
        3) Возвращает: {"attachment": "video{owner_id}_{video_id}"}
        Видео обрабатывается VK асинхронно — после загрузки оно может быть некоторое время недоступно.
        Это нормально, wall.post всё равно сработает — VK просто покажет процесс обработки в посте.
        """

    def post_to_wall(self, message: str, attachments: list[str] = None,
                     publish_date: int = None) -> dict:
        """
        Публикует пост на стене сообщества.
        wall.post(owner_id=-group_id, from_group=1, message=..., attachments=..., publish_date=...)
        publish_date — Unix timestamp для отложенной публикации СРЕДСТВАМИ САМОГО ВК
        (используем его НЕ для отложенной публикации — её делаем своим scheduler-ом).
        Возвращает: {"vk_post_id": int, "vk_post_url": str}
        """
```

**Важные нюансы:**
- При ошибках VK API (`error.error_code`) — выбрасывать понятное исключение `VKAPIError` с русским описанием (например: «Токен ВК истёк. Получи новый и пропиши в .env»).
- Все вызовы логировать в `ai_calls.jsonl` с `provider: "vk"`, `request_type: "upload_photo|upload_video|wall_post"`.
- Если в `.env` нет `VK_COMMUNITY_TOKEN` или `VK_GROUP_ID` — публикация недоступна, в UI кнопки публикации серые с тултипом «Настрой VK в .env (см. README)».

### D3. Эндпоинты публикации

```
POST /api/posts/<id>/publish-now
    → Тело: {} (всё берём из БД по id)
    → 1. Валидация: text не пуст, длина <= 4096, есть ли media
    → 2. Если есть image — VK upload_photo
       Если есть video — VK upload_video
    → 3. wall.post с attachments
    → 4. Обновить Post: status='published', vk_post_id, vk_post_url, published_at=now()
    → Ответ: {success: true, vk_post_url: "..."}

POST /api/posts/<id>/schedule
    → Тело: {scheduled_at: "2026-05-15T09:00:00Z"} (ISO 8601, UTC)
    → 1. Валидация: время в будущем (как минимум +5 минут от now), пост валиден
    → 2. Сохранить в БД: status='scheduled', scheduled_at=...
    → 3. Зарегистрировать задачу в APScheduler с id=`publish_post_{post_id}`
    → Ответ: {success: true, scheduled_at: "...", post_id: ...}

DELETE /api/posts/<id>/schedule
    → Отмена расписания
    → 1. Удалить задачу из APScheduler
    → 2. Установить status='draft', scheduled_at=NULL
    → Ответ: {success: true}

GET /api/posts?status=scheduled
    → Список запланированных постов с их временами

GET /api/posts/<id>/publication-history
    → История публикаций этого поста (если опубликован — одна запись с vk_post_url)
```

**Расширение модели `Post`:**
- `scheduled_at` (datetime, nullable) — когда запланировано
- `published_at` (datetime, nullable) — когда фактически опубликовано
- `vk_post_id` (str, nullable) — id поста в ВК после публикации
- `vk_post_url` (str, nullable) — прямая ссылка вида `https://vk.com/wall-{group_id}_{post_id}`
- `last_publish_error` (text, nullable) — последняя ошибка публикации, если была

Статусы (`status`): `draft` | `scheduled` | `published` | `publish_failed`.

### D4. Планировщик (`core/scheduler.py`)

APScheduler с `BackgroundScheduler` запускается в `app.py` при старте Flask. SQLAlchemy job store на той же `lora.db` (отдельная таблица — APScheduler сам создаст).

При старте приложения:
1. Загрузить все посты со `status='scheduled'` и `scheduled_at > now()` — переcоздать задачи (на случай рестарта).
2. Все посты со `status='scheduled'` и `scheduled_at <= now()` (пропущенные, например из-за того что сервер был выключен) — выполнить публикацию немедленно. **Это важная логика устойчивости** — пользователь должен быть уверен что отложенная публикация не потеряется.

**Функция-исполнитель задачи:**

```python
def publish_scheduled_post(post_id: int):
    """
    Вызывается APScheduler-ом в назначенное время.
    1) Загружает Post по id, проверяет status == 'scheduled'
    2) Вызывает VKClient.upload_* + post_to_wall
    3) Обновляет Post: status='published' или 'publish_failed' + last_publish_error
    4) Логирует результат в ai_calls.jsonl
    5) НЕ удаляет задачу — APScheduler сам удалит её после выполнения (у задач date trigger)
    """
```

При ошибке публикации — НЕ повторяем автоматически, ставим `status='publish_failed'`, в библиотеке такие посты выделяются цветом и кнопкой «Повторить публикацию».

### D5. UI расписания

В редакторе поста — две кнопки рядом:
- **📤 Опубликовать сейчас** — модалка подтверждения → POST publish-now → toast «Пост опубликован: [ссылка]»
- **🕐 Запланировать** — открывается мини-модалка с date-time picker (нативный `<input type="datetime-local">` достаточно для v0.2; локаль RU; минимум — текущее время + 10 минут) → POST schedule → toast «Запланировано на {datetime}»

В библиотеке для статусов:
- `draft` — серая плашка
- `scheduled` — горчичная плашка с временем «Через 2 часа» (относительный формат через JS-функцию)
- `published` — зелёная плашка «Опубликовано {дата}» с иконкой-ссылкой 🔗 на пост в ВК (открывается в новой вкладке)
- `publish_failed` — красная плашка «Ошибка публикации: {краткое сообщение}» + кнопка «Повторить»

Фильтр по статусу в библиотеке расширяется этими новыми значениями.

### D6. «Создать на основе» (повторное использование постов)

В карточке любого поста (особенно опубликованного) — кнопка **📋 Создать на основе**.

Логика:
- Создаётся НОВЫЙ Post с теми же `text_content`, `image_path` (или `video_path`), `rubric_key`, `topic`, **но** `status='draft'`, `scheduled_at=NULL`, `published_at=NULL`, `vk_post_id=NULL`, `vk_post_url=NULL`.
- Открывается в редакторе.
- В UI помечается «📋 Создан на основе поста #{old_id} от {date}».

Эндпоинт: `POST /api/posts/<id>/duplicate` → возвращает `{new_post_id}`.

## 11. Правила работы

- **Ничего сверх scope v0.2.** Соблазны — в раздел «Идеи на v0.3» в отчёте сессии.
- **Перед нетривиальной командой** — одна строка объяснения на русском.
- **VK API нюансы:** обязательно через web search перед написанием `vk_client.py` подтверди актуальные эндпоинты, версию API (`5.199` на момент написания этого ТЗ), коды ошибок и формат attachments. ВК периодически вводит ограничения и меняет требования к токенам.
- **Если что-то блокирует** (ошибка VK 15 «access denied», нет нужных прав в токене, ошибка APScheduler в Windows) — остановись, опиши проблему пользователю с конкретным предложением что нужно от него, не пытайся обойти молча.
- **APScheduler на Windows** имеет нюансы с фоновым процессом — убедись что приложение корректно закрывается через Ctrl+C и не оставляет зомби-процессов.
- **Время:** все datetime в БД хранить в UTC, в API принимать ISO 8601, в UI отображать в локальном времени пользователя через JS (`new Date(iso).toLocaleString('ru-RU')`).
