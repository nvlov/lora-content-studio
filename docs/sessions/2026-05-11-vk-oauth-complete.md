## Хронология

1. **Цель сессии (по `vk-oauth-004`):** довести OAuth до конца, медиа-публикация подтверждена в реальности.
2. **Vk-oauth-004 Шаг 2:** OAuth flow с новым приложением `54587881` (профиль самозанятого) — VK на экране авторизации показал только «Общая информация», расширенные scope не появились. → 🔴 RED.
3. **Доп. исследование:** официальная документация `id.vk.ru` и ответ поддержки VK (зафиксирован на vc.ru) — **«расширенные API-доступы больше не выдаются»** (политика VK с 2025 года). Новые приложения-третьих-сторон не могут получать scope `wall`/`photos`/`video` ни при какой верификации.
4. **Попытка community-token workaround:** `photos.getMessagesUploadServer` + `photos.saveMessagesPhoto` отдают upload_url, фото грузится, save возвращает корректный photo-attachment ID с `access_key`. Первая публикация (`vk.com/wall-237689862_4`) — фото не прикрепилось. Гипотеза: отсутствует `access_key` в attachment-строке. Вторая публикация (`vk.com/wall-237689862_5`) с `access_key` — **фото всё равно не прикрепилось**.
5. **Расширенный probe** всех photo-upload методов: единственный, который отдаёт `upload_url` для community-токена — `photos.getMessagesUploadServer` (фото в системном альбоме `-64` «сохранённые из сообщений»). Эти фото VK конструктивно не даёт прикреплять к wall.post — by design платформы, не баг. Все остальные методы (`photos.getWallUploadServer`, `photos.getUploadServer`, `photos.createAlbum`, `photos.saveWallPhoto`, `docs.getWallUploadServer`, `video.save`) — code 27 «method is unavailable with group auth».
6. **Вердикт диагностики:** **в текущем VK API нет рабочего пути для community-токена прикрепить фото к wall.post**. Не только видео (это было известно из v0.2), но и фото.

## Продуктовое решение

После двух подряд провальных smoke-тестов Dr. Nik принял решение **полностью убрать публикацию медиа из проекта**:

> «согласен, идем к варианту только генерация и публикация текста. изображение и видео полностью убираем из проекта. UX/UI изменим позже, сейчас только фиксация состояния, запись логов»

## Что сделано в коде (бэкенд)

1. **`core/vk_client.py`** — полная переработка:
   - Удалены методы `upload_photo()` и `upload_video()`.
   - Удалены capability-флаги `media_publish_supported()` / `video_publish_supported()`.
   - `post_to_wall()` упрощён: больше не принимает `attachments` (только `message`).
   - `_VK_ERROR_HINTS` почищен от устаревших медиа-кодов (27, 1051) — оставлены только релевантные для текстовой публикации (5, 9, 14, 15, 100, 214).
   - Docstring модуля обновлён: «публикация только текстовых постов».
   - **OAuth 2.1 инфраструктура оставлена нетронутой** как «спящая заготовка» (`_ensure_fresh_token`, OAuth fields в `__init__`) — на случай если VK сменит политику по media scope.

2. **`core/scheduler.py`** — `_do_publish()`:
   - Удалён блок сбора `attachments` из `post.media_kind`/`image_path`/`video_path`.
   - Всегда вызывает `vk.post_to_wall(message=post.text_content)` без attachments.
   - Поля media в модели `Post` оставлены для backward-compat (старые черновики не ломаются).

3. **`routes/api.py`** — `/api/vk/status`:
   - Удалены поля `media_publish_supported` и `video_publish_supported` из ответа.

4. **`tests/test_vk_oauth.py`**:
   - Удалены ассерты по `media_publish_supported`/`video_publish_supported`.
   - 20/20 тестов проходят.

## Что НЕ сделано (отложено)

- **UI / фронтенд** — не тронут. Кнопки «Сгенерировать через Kling», вкладка «Медиа», медиа-библиотека, прикрепление файлов к посту — всё остаётся в HTML/JS как есть. Удаление этого функционала — отдельная задача (по решению Dr. Nik «UX/UI изменим позже»).
- **Kling-интеграция в бэкенде** — оставлена (она генерирует картинки, которые можно сохранить локально и использовать вне VK).
- **Поля `image_path` / `video_path` / `media_kind` в модели Post** — оставлены для backward-compat. Не используются при публикации.
- **Удаление двух тестовых постов (`wall-237689862_4` и `_5`)** в группе `lora_ai_english` — делает Dr. Nik вручную.

## Ключевой вывод проекта (для архива)

**VK API в 2026 не предоставляет третьим лицам путей публикации медиа в сообщество.** Ни VK ID OAuth 2.1 (policy block), ни community-token (technical block через album_id=-64), ни service-token, ни ключи защищённого приложения. Единственная рабочая медиа-публикация в VK сейчас — **через UI самого VK** (drag-n-drop, мобильное приложение, VK Studio). Это решение VK как платформы, не наша архитектурная ошибка.

Lora Content Studio пивотится в инструмент «генерация контента (текст + опционально картинка для ручной выгрузки) + автоматическая публикация текста в VK». Это рабочий продукт, который закрывает половину pain-а (генерация и автопостинг текста), а медиа-приложение остаётся ручной операцией Dr. Nik (~30 секунд после публикации текста).
