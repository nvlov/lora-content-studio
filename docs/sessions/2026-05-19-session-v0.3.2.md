## Ключевые решения сессии

1. **Workflow медиа — внешний, не direct API.** Приложение только формирует промпт. Пользователь идёт в Kling Web UI вручную и загружает результат обратно. Direct API через `KlingImageProvider` оставлен как dormant code для возможного будущего использования.
2. **Звук в видео: только английский, только Kling 3.0 Omni.** Русская речь через LipSync (двухэтапный flow с TTS) — отложено в backlog. Зафиксировано в memory.
3. **Связь промпт↔результат на уровне БД, не только UI.** `MediaAsset.source_prompt_id` — FK. Это даст возможность в будущем (Вариант C из обсуждений) сделать автомодификацию промпта на основе истории оценок.
4. **Notion-стиль без кремового.** Брендовые цвета остаются только в логотипе. Контент-превью VK-постов остаётся брендовым в самом VK (на нашей стороне приложение нейтрально-белое).
5. **Поле `negative_prompt`** — отдельное в Claude-ответе, отдельная кнопка «Копировать negative» в UI. Не объединяется с основным промптом, потому что Kling Web UI принимает их отдельно.
6. **Markdown-фильтр** для постов в VK реализован в `routes/api.py:_strip_vk_unfriendly_markdown`. НЕ в `llm_client.generate()` — это важно, потому что `prompt_generator.py` ждёт JSON в ответе Claude и фильтр мог бы испортить JSON-строки.

## Возникшие проблемы

- **kling.ai отдаёт HTTP 446** для WebFetch (анти-бот защита). Обошёл через WebSearch + зеркала (fal.ai, klingapi.com, atlascloud blog). Один Discovery-агент в фоне закрыл основную часть фактов, второй прогон WebFetch+WebSearch уточнил по 3.0 Omni и LipSync.
- **Discovery вернул вердикт YELLOW** по 3 unknowns (схема `image_reference`, поддержка `negative_prompt` в kling-v1, точный max prompt size). Но т.к. workflow внешний (без direct API), эти unknowns не блокируют. Это перевело вердикт для нашего scope в GREEN. Явно зафиксировано в чате до начала кодирования.
- **JS-рефакторинг был большим** (≈900 строк нового кода). Сделано через `Write` (полная перезапись), не через серию `Edit` — чище и быстрее.
