### Публикаторы

**Базовый интерфейс** `core/publishers/base.py`:
- `BasePublisher` — абстрактный класс с методами `is_configured`, `publish_text`, `publish_with_image`, `publish_with_video`
- `PublishResult` (TypedDict) — единый формат возврата: platform/post_id/post_url
- `PublishError` — общее исключение, наследуемое от Exception

**Реестр** `core/publishers/__init__.py`:
- `get_publisher(name)` — фабрика с ленивым импортом
- `available_publishers()` — список со статусом каждого

**VK** — `VKClient` + адаптер `VKPublisher`, тонкая обёртка. `VKAPIError` теперь наследуется от `PublishError` для общего ловителя в scheduler/CLI.

**Telegram** — полная реализация Bot API через httpx:
- `sendMessage` (текст до 4096)
- `sendPhoto` + caption (до 1024) — для будущей текст+картинка
- `sendVideo` + caption (до 1024) — для будущей текст+видео
- Без `TELEGRAM_BOT_TOKEN` и `TELEGRAM_CHANNEL_ID` отвечает «не настроен» — UX-понятно.
- URL поста собирается из `chat.id` + `message_id`. Для приватных каналов `https://t.me/c/<short>/<id>`.

**YouTube / TikTok** — placeholder-классы с `IMPLEMENTED = False` и docstring-планом подключения (что нужно: Google Cloud / TikTok Developer, OAuth, app review). Не работают, но не падают при загрузке.

## Ключевые решения

1. **Не подключаемся к utopy.ai.** Discovery показал: это российский веб-агрегатор без публичного API, без MCP, без Skill. ProxyAPI у нас уже даёт доступ к тем же моделям (gpt-image-2, Kling) через нормальный REST. Selenium-обход не рекомендую — ToS-violation, нестабильно.

2. **Telegram — реальная реализация, не stub.** Несмотря на отсутствие канала сейчас, модуль написан с полной логикой Bot API. При появлении токена работает сразу без правок кода. Заглушка — только в смысле что ключей в `.env` нет.

3. **YouTube/TikTok — placeholder, не реализация.** OAuth-flow для них слишком тяжёлый чтобы делать вслепую: требуют developer account, app review (особенно TikTok), что без реальной потребности — потеря времени. Документация плана в docstring модуля.

4. **`Post.target_platforms` как JSON-text vs отдельная таблица.** Выбрал JSON-колонку — проще для текущей задачи (мультиплатформенный sched), SQLite это терпит. Если понадобится индекс «найди все посты для Telegram» — добавим отдельную таблицу.

5. **`PostPublication` отдельной таблицей, а не дублированием полей в Post.** Чтобы добавление новой платформы не требовало миграции схемы. Legacy `vk_post_id`/`vk_post_url` в Post — для backward-compat с UI; со временем UI можно мигрировать на `post_publications`.

6. **VKAPIError наследуется от PublishError.** Это даёт `except PublishError` в scheduler/CLI работающий и для VK-специфичных ошибок, не ломая существующий `except VKAPIError` в routes/api.py.

7. **Тег v0.3.3 ставится ДО рефакторинга.** Чистая точка отката. Если в v0.4.0 что-то сломалось — `git checkout v0.3.3` возвращает к рабочему состоянию.

8. **Бренд-ассеты вне `core/`.** 75 МБ оригиналов PNG — это не код. `assets/lora/` корневая папка снимает 75 МБ концептуально с кодовой части. PNG в `.gitignore`, оптимизированные JPG (~1 МБ) в репо как production-ассеты.

## Возникшие проблемы

- **Edit-инструмент требует Read перед редактированием перемещённых файлов** — после `git mv` файлы считаются «новыми». Пришлось делать Read для каждого затронутого файла перед Edit.
- **Невидимые символы в `.env.example`** — первая попытка Edit с многострочным шаблоном не нашла совпадение, видимо из-за невидимых байтов. Пересчитал по короткому уникальному фрагменту.
- **Кириллица в Windows-консоли** — sys.stdout по умолчанию cp1251, в Python 3.9+ можно `reconfigure(encoding='utf-8')`. Добавил в `manage.py` блок реконфигурации.

### Урок: ProxyAPI не догоняет Anthropic — гибрид необходим

ProxyAPI как российский прокси отстаёт от свежих релизов Anthropic.
Это не баг ProxyAPI, а структурная задержка любого прокси. Решение:
- Тексты пишет Claude в чате (через Skill, у меня всегда свежая модель)
- Картинки/видео — через ProxyAPI (там модели OpenAI/Kling, проверены)

Это снижает зависимость от одной точки отказа и даёт доступ к самой
свежей версии Claude для генерации контента.
