## Ошибки и фиксы (по ходу)

### Ошибка #1 — VK code 27 (Сценарий B): `photos.getWallUploadServer` недоступен с community-token

**Где:** загрузка фото на стену сообщества.

**Что именно:**
```
VK API (upload_photo): Group authorization failed: method is unavailable with group auth. [code=27]
```

**Корневая причина:** документированное ограничение VK API — методы `photos.getWallUploadServer` и `video.save` для стены сообщества **недоступны с community access token**. Они требуют **user access token** админа группы со скоупом `photos, video, wall, manage`.

В README v0.2 community-token был указан как основной (через «Управление → Работа с API → Создать ключ»), а user-token через OAuth — как «альтернативный». Оказалось наоборот: для постов с медиа user-token обязателен; community-token годится только для текстовых постов (как в сценарии A).

**Это не баг кода** — методы в `core/vk_client.py` правильные. Фикс — в документации и инструкции по получению токена.

**Решение (предложение):**
1. Doc-фикс в README (≤ 15 строк): сделать user-token основным путём, community-token упомянуть как fallback для text-only автоматизации.
2. `.env.example` — комментарий «нужен user access token админа группы».
3. В `core/vk_client.py` — добавить понятную русскую подсказку для code 27 в `_VK_ERROR_HINTS`. Это 1 строка.

Код `vk_client.py`/`scheduler.py`/`api.py` менять не нужно — токен прозрачно проксируется в VK, какого бы типа он ни был.

### Ошибка #2 — `_parse_iso_utc` уносил время в локальную TZ (затронуло E/F)

**Где:** `routes/api.py`, функция `_parse_iso_utc`.

**Что именно:** при schedule на UTC `2026-05-10T14:45:16Z` бэкенд сохранял `2026-05-10T17:45:16Z` (Moscow time, naive). Задача оказывалась на 3 часа позже желаемого.

**Корневая причина:** `dt.astimezone(tz=None).replace(tzinfo=None)` — `tz=None` для `astimezone` означает «локальная зона», а не UTC. Затем `.replace(tzinfo=None)` отбрасывает tzinfo, и naive datetime сохраняется как будто это UTC (в коде везде так и трактуется).

**Фикс (4 строки):** заменено на `dt.astimezone(timezone.utc).replace(tzinfo=None)` — корректная конвертация в UTC, потом naive UTC.

Закоммичу отдельно как `fix(api): correct UTC conversion in _parse_iso_utc (vk-launch-001)`.

**Затрагивает ли UI v0.2?** Да. Frontend `app.js` отправляет `localDate.toISOString()` — это всегда UTC с Z. Тот же баг сработал бы и через UI. Хорошо что нашли.

## Решение

**v0.2.2 после фиксов готов для:**
- ✅ Ручной публикации текстовых постов (Сценарий A)
- ✅ Запланированной публикации текстовых постов с восстановлением задач после рестарта (Сценарии E, F)
- ❌ Постов с медиа (B, C, D) — пока пользователь не получит **user-token** через OAuth-flow вместо community-token

**Что сделано в этой ветке (vk-launch/v0.2.1-validation):**
1. `routes/api.py` — фикс таймзоны в `_parse_iso_utc` (4 строки): `astimezone(tz=None)` → `astimezone(timezone.utc)`. Без этого ВСЕ scheduled-публикации уезжали в `now + локальная_TZ_offset`.
2. `core/vk_client.py` — добавлен hint для VK code 27 в `_VK_ERROR_HINTS` (1 строка): «Метод недоступен с community-токеном. Для постов с медиа нужен USER-токен админа».
3. `core/vk_client.py` — обновлены тексты hint-ов 5, 7, 15: упоминают user-token и OAuth.
4. `README.md` — раздел «Настройка VK API» переписан: user-token через OAuth теперь основной путь, community-token явно помечен как недостаточный для медиа.
5. `.env.example` — комментарий к `VK_COMMUNITY_TOKEN` уточнён: «кладёшь сюда именно user-токен».

Имя переменной `VK_COMMUNITY_TOKEN` оставлено для совместимости (переименование = рефакторинг, не делал — это запрещено разделом 7 ТЗ).

## Что нужно от пользователя для разблокировки B/C/D

1. На [dev.vk.com](https://dev.vk.com) убедиться что есть Standalone-приложение, записать `App ID`.
2. Открыть в браузере (подставив свой `APP_ID`):
   ```
   https://oauth.vk.com/authorize?client_id={APP_ID}&display=page&scope=wall,photos,video,stories,offline,manage&response_type=token&v=5.199&redirect_uri=https://oauth.vk.com/blank.html
   ```
3. Авторизоваться **под личным аккаунтом** (тот, что админ группы).
4. Скопировать `access_token=vk1.a...` из адресной строки после редиректа.
5. Заменить значение `VK_COMMUNITY_TOKEN=` в `.env` на этот user-токен.
6. Перезапустить Flask.
7. Запросить второй прогон `vk-launch-002` для сценариев B, C, D.
