# Hotfix 001 — Модалка медиа-библиотеки зависала при загрузке

**Дата:** 2026-05-10
**Версия после фикса:** v0.2.1
**Длительность:** ~15 минут (Claude Code, по `tasks/CLAUDE_CODE_HOTFIX_001.md`)

## Симптом
При открытии главной страницы сразу всплывала пустая модалка «Выбор медиа из библиотеки», блокировала весь интерфейс. Кнопка «Закрыть» и `Esc` не реагировали.

## Корневая причина
CSS-каскад. В `static/css/style.css` (v0.2) правило `.modal-overlay { display: flex; ... }` имеет ту же специфичность (0,0,1,0), что и встроенное `[hidden] { display: none }` из user-agent stylesheet, но идёт ПОЗЖЕ — поэтому побеждает. Атрибут `hidden` в HTML фактически игнорировался.

То же справедливо для `.kling-progress[hidden]` (тоже `display: flex`).

Кнопка «Закрыть» picker-а не работала вторично: её `onclick` навешивается **внутри** функции `openMediaPicker()`, которая ни разу не вызывалась — модалка показалась чисто из-за CSS, без участия JS, поэтому обработчики `picker-close.onclick` и сетка `picker-grid` оставались пустыми.

## Что изменилось
- `static/css/style.css`, +2 строки после правила `.modal-overlay`:
  ```css
  /* hotfix-001: HTML-атрибут [hidden] должен побеждать display:flex/.modal-overlay/.kling-progress */
  .modal-overlay[hidden], .kling-progress[hidden] { display: none; }
  ```
  Специфичность 0,0,2,0 — побеждает 0,0,1,0. Без `!important`, без рефакторинга.

- `static/js/app.js`, +5 строк в существующий `keydown`-обработчик: Esc теперь закрывает `#picker-overlay` (у `confirmModal` свой собственный обработчик Esc, его не трогали). Это требование Шага 3 ТЗ — «убедиться что Esc тоже закрывает модалку».

Итого 7 строк изменений, в двух файлах. Никаких правок в Python, HTML, моделях, конфигах.

## Гипотезы которые НЕ подтвердились
- **H1 (JS-ошибка при init).** В Console только accessibility-warning «label isn't associated with form field», JS не падает.
- **H3 (бэкенд `/api/media` валится).** В `server_log.png` сервер чист, scheduler запустился, ошибок нет. На Network скриншоте запрос к `/api/media` ВООБЩЕ не делался — что подтверждает: проблема на этапе рендера, ещё до JS-вызова.
- **H4 (опечатка в селекторе).** Селектор `#picker-close` совпадает между HTML и JS. Дело не в селекторе, а в том, что код навески обработчика просто не выполнялся.

Ближе всего к правде была **H2** (модалка показывается случайно при загрузке) — но не из-за ошибочного JS-вызова, а из-за CSS-каскада.

## Потенциальные риски, замеченные по дороге (на v0.3 или отдельный hotfix)
- **Anti-pattern `display:flex` + `[hidden]` для toggle-видимости.** Сейчас закрыли двумя селекторами; если в будущем появится ещё один блок с `display: flex` и атрибутом `hidden`, он снова сломается. Системно правильнее — использовать класс `.is-open`, а не атрибут `hidden` для контролируемых через CSS оверлеев. Зафиксировать в v0.3.
- **Accessibility-warning «label isn't associated with form field»** в Console. Не критично для работы, но стоит навесить корректные `for=...` / обернуть инпут в `<label>` в `_template.md` стиле — особенно `#topic`, `#post-text`, `#prompt-idea`.
- **`openMediaPicker` не имеет таймаута / обработки ошибки сети** — если `/api/media` упадёт, в гриде останется «Загрузка…». Не покрыто этим hotfix.
