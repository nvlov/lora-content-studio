# Обзор проекта и план развития — Lora Content Studio

**Дата обзора:** 2026-05-10
**Текущая версия:** v0.1.0 (MVP, принято 2026-05-08)
**Тип отчёта:** Резюме состояния + план развития (без изменений в коде)

---

## 1. Резюме состояния проекта

### 1.1. Назначение
Локальный инструмент для подготовки контента в сообщество ВК «Lora»: генерация текстов (Claude Sonnet 4.5 через ProxyAPI) и изображений (Kling AI), хранение и редактирование черновиков, превью в стиле VK.

### 1.2. Стек
- **Backend:** Python 3.10+, Flask 3.x, SQLAlchemy + SQLite (`data/lora.db`)
- **HTTP:** `httpx`
- **Изображения:** Pillow (валидация/ресайз)
- **Frontend:** чистый HTML/CSS/JS, без сборки и фреймворков
- **Шрифты/бренд:** Lora (заголовки) + Inter (тело), палитра «крем + горчица + тёмно-зелёный»
- **Логи:** `logs/app.log` (rotating), `logs/ai_calls.jsonl` (аудит вызовов AI)

### 1.3. Архитектура
```
app.py → routes/{pages,api}.py → core/{db,models,llm_client,image_clients,prompts,logging_utils}.py
templates/index.html  +  static/{css/style.css, js/app.js}
```
- Чёткое разделение слоёв: конфиг → модели → сервисы → роуты → фронт
- SPA с тремя табами (`Создание поста / Библиотека / Настройки`), переключение через `display`
- Все API возвращают JSON, ошибки — `{error: "..."}` с понятным русским текстом

### 1.4. Реализованные фичи (v0.1)
- 4 рубрики со своими системными промптами (Слово дня, Типичная ошибка, Фраза из жизни, Грамматика)
- Генерация текста (Claude) и изображений (Kling, JWT HS256, polling)
- Ручная загрузка изображения (jpg/png/webp, до 10 МБ, авто-ресайз)
- Превью поста в стиле VK
- CRUD черновиков, поиск + фильтр по рубрике в библиотеке
- Редактирование промптов и шаблонов изображений в табе «Настройки»
- Логирование всех AI-вызовов (токены, статус, ошибка)
- Bezopasная обработка ошибок API (без stack-trace наружу)

### 1.5. Сильные стороны
- Полностью завершённый scope v0.1, рабочее состояние
- Аккуратная архитектура и читаемый код, нет лишних зависимостей
- Брендовая консистентность (типографика, палитра, скругления, тени)
- Возможность редактировать промпты без перезапуска / правки кода
- Журнал сессий и документация на русском

### 1.6. Известные ограничения (by design v0.1)
- Только desktop (≥1280px), нет адаптива
- Нет авторизации (single-user локальный инструмент)
- Нет публикации в VK, расписания, видео, аналитики
- Нет soft-delete, undo, истории правок
- Генерация изображения 1–2 минуты (особенность Kling API)

### 1.7. Слабые места и риски
- **UX-обратная связь:** во время длительных операций (Kling 1–2 мин) только текст в кнопке — нет прогресс-индикатора, прогноза времени, возможности отмены
- **Доступность:** нет ARIA-атрибутов, фокус-стилей, проверки контраста, навигации с клавиатуры
- **Безопасность XSS:** текст из API вставляется в превью — нужно убедиться, что везде используется `textContent`/`innerText`, а не `innerHTML`
- **Тесты:** отсутствуют (даже smoke-тесты роутов)
- **Кастомный JWT:** ручная реализация HS256 — стоит заменить на `pyjwt`
- **Один большой `index.html` + один `app.js` (~420 строк):** при росте функциональности (v0.2+) станет тяжело сопровождать
- **`routes/api.py` (242 строки):** при добавлении эндпоинтов потребуется разбиение
- **Нет миграций БД** (Alembic) — изменение схемы потребует ручных шагов
- **Нет резервного копирования** SQLite и `static/uploads`

## 2. План развития

План разбит по приоритету и привязан к запланированным версиям из `PROJECT_LOG.md`. Внутри каждой версии — задачи отсортированы по соотношению «эффект / трудоёмкость».

### 2.1. Версия v0.1.x — UX/UI-полировка и техдолг (ближайшие 1–2 недели)

Цель: убрать главные «острые углы» v0.1, повысить ощущение надёжности и качества. Без новых фич.

**A. Обратная связь и состояния (high impact)**
1. **Skeleton-loader** для библиотеки и плейсхолдер «Пока пусто, создайте первый пост → [кнопка]» во вьюхе создания.
2. **Прогресс-индикатор Kling**: спиннер + таймер «Генерация… 0:42 / ~1:30», блокировка кнопок, возможность «Отменить» (минимум на фронте).
3. **Toast-система**: единая позиция (правый-нижний угол), типы success/warn/error, авто-скрытие, очередь.
4. **Состояние «несохранённые изменения»**: индикатор «●» возле кнопки «Сохранить», подтверждение при уходе со вкладки.
5. **Диалог подтверждения удаления** черновика (сейчас удаление мгновенное).

**B. Доступность и качество интерфейса**
6. ARIA-роли табов (`role="tablist"`, `aria-selected`), фокус-кольца, навигация Tab/Enter/Esc, hot-keys (`Ctrl+Enter` — сгенерировать, `Ctrl+S` — сохранить).
7. Аудит контраста (WCAG AA) для текста на крем-фоне и горчичных кнопках.
8. `prefers-reduced-motion` для анимаций.
9. Lang-атрибут `ru`, корректные `<label for>` у всех инпутов.

**C. Превью и редактор**
10. **Полноэкранный превью** изображения по клику.
11. **Подсчёт символов** в тексте (с лимитом VK 4096) + цветовая индикация переполнения.
12. **Markdown→VK-форматирование** (минимум: переносы строк, эмодзи-пикер).
13. **«Перегенерировать» с сохранением темы** и «Уточнить» (продолжить через follow-up к Claude).

**D. Библиотека**
14. Сортировка (по дате, рубрике).
15. Пагинация / lazy-load при ≥30 черновиках.
16. Тэги/статусы: `черновик / готов / опубликован` (поле `status` уже есть в модели — задействовать).
17. **Дублирование черновика** («Создать на основе»).

**E. Настройки**
18. Превью «как будет работать» — мини-эмулятор подстановки `{topic}` с примером.
19. Кнопка «Откатить к промпту по умолчанию».
20. Версионирование промптов (хранить последние 3 версии, чтобы можно было откатиться).

**F. Техдолг**
21. **XSS-аудит** `app.js` — все вставки текста через `textContent`.
22. Заменить ручной JWT на `pyjwt`.
23. Разбить `routes/api.py` на `routes/api/{posts,rubrics,generation,settings}.py`.
24. Добавить **Alembic** для миграций.
25. Скрипт **бэкапа** `data/lora.db` + `static/uploads` (PowerShell под Windows, по расписанию).
26. **Smoke-тесты** pytest: каждый эндпоинт возвращает ожидаемый статус; mock внешних API.
27. **Pre-commit hooks**: `ruff`, `black`, базовая проверка `.env` не закоммичен.

### 2.2. Версия v0.2 — публикация в VK и контент-план

Цель из `PROJECT_LOG.md` + конкретизация.

1. **Интеграция VK API** — публикация поста (текст + 1 изображение) в стену сообщества; OAuth-токен в `.env`.
2. **Расписание** — APScheduler, поле `scheduled_at` у поста, фоновый воркер.
3. **Календарь на месяц** — новый таб «Календарь»: сетка 7×N, drag-and-drop черновиков на даты.
4. **Шаблоны рубрик «по дням недели»** — например, «понедельник = Слово дня».
5. **История публикаций** — журнал с ссылкой на пост в VK и статусом (опубликовано/ошибка).
6. **Двойной препросмотр**: рядом с VK-видом — мобильный вид VK (важно, т.к. >70% аудитории мобильные).
7. **Превью карусели** (если v0.2 включит мульти-изображения).

### 2.3. Версия v0.3 — видео и аналитика

1. **Kling Video** — короткие ролики (5–10 сек) для рубрик; статус генерации, превью.
2. **Аналитика VK** — собирать охваты/лайки/комментарии по опубликованным постам через VK API; график «эффективность по рубрикам».
3. **A/B-тестирование промптов** — генерация 2 вариантов текста, выбор лучшего с фиксацией метрик.
4. **Подсказки на основе истории** — «посты этой рубрики набирают больше лайков, если короче ~600 знаков».

### 2.4. Версия v0.4 — деплой и устойчивость

1. **Docker + docker-compose** (Flask + SQLite-volume + nginx-reverse-proxy).
2. **Деплой на VPS** + Caddy/nginx, HTTPS, basic-auth (single-user).
3. **GitHub Actions:** lint + тесты + сборка образа.
4. **Миграция SQLite → PostgreSQL** при росте (опционально).
5. **Sentry** для ошибок, **OpenTelemetry** для метрик AI-вызовов.

## 3. Сквозные UX/UI-принципы (рекомендация на все версии)

1. **«Видимость состояния системы»** (Nielsen #1) — каждая длительная операция должна иметь индикатор и оценку времени.
2. **«Контроль и свобода пользователя»** (Nielsen #3) — отмена, undo, подтверждение деструктивных действий.
3. **Консистентность** — одна toast-система, один стиль кнопок, единые radius/тени (уже хорошо).
4. **Мобильное превью важнее десктопного** для контента в VK.
5. **Скорость > красоты** — оптимистичные UI-обновления (PATCH черновика — обновляем UI сразу, откатываем при ошибке).
6. **Минимум кликов** для повторяющегося сценария «сгенерировать → подправить → сохранить → опубликовать».
7. **Контекстная помощь** вместо отдельной документации (тултипы, плейсхолдеры с примерами).

## 5. Метрики, по которым стоит мерять прогресс

- Время от «открыть приложение» до «сохранён готовый пост» (целевое: <3 мин для текстового, <5 мин с картинкой).

- Количество ошибок Kling/Claude в `ai_calls.jsonl` за неделю.
- Размер `static/uploads` и БД (мониторить рост).
