## 2. Главные принципы (в порядке приоритета)

### 2.1. Discovery first, всегда

**Никаких архитектурных решений до того как факты подтверждены.** Это правило №1.

Перед написанием ТЗ на любую интеграцию с внешним сервисом или новую крупную фичу:
1. Web search актуальной документации (фильтр года, не старее 12 месяцев)
2. Поиск свежих кейсов разработчиков с тем же use case (Habr, форумы, GitHub Issues)
3. **Самый дешёвый эксперимент** который может опровергнуть план — делается **первым**
4. Результат — файл `docs/<integration>-research.md` с вердиктом GREEN / YELLOW / RED

**YELLOW = STOP, НЕ GO.** Если есть незакрытые unknowns — реализация не начинается. Сначала закрываются unknowns малыми экспериментами (каждый ≤30 минут), потом GREEN, потом код.

### 2.2. Prototyping over architecture

Перед написанием 200 строк production-кода — 30 строк прототипа. «Сначала curl в терминале или вкладка браузера, потом Python».

**Правило 80/20 в Discovery:** 30 минут эксперимента в браузере = 5 дней переделок кода. Всегда выбирайте 30 минут.

Примеры дешёвых экспериментов которые могли бы сэкономить дни в кейсе VK:
- Открыть OAuth-URL в браузере → посмотреть какие права запрашиваются → 5 минут
- Сделать `account.getAppPermissions` через curl с тестовым токеном → 10 минут
- Прочитать 3 свежих Habr-комментария с ключевыми словами «VK ID scope vkid.personal_info» → 15 минут

Любой из этих экспериментов до Phase 2 показал бы что нужен бизнес-профиль ИП/самозанятого/юрлица.

### 2.3. Honest uncertainty

В каждом совете явно различать:
- **«Знаю точно»** — есть прямой источник 2025–2026 года или работающий пример
- **«Предполагаю»** — основано на общих знаниях, требует проверки
- **«Не знаю»** — нужен web search или эксперимент

**Запрещённые формулировки в этом проекте:**
- «Я думаю что VK позволяет…» (без проверки)
- «Раньше работало так…» (без проверки актуальности)
- «По умолчанию это работает…» (без проверки документации)
- «Должно сработать…» (без эксперимента)

**Корректные формулировки:**
- «По документации VK от <дата> это работает так — но не проверял на нашем случае, давайте сделаем минимальный тест»
- «Я не знаю текущего состояния API, нужно web search»
- «Это предположение, его опровергнет один curl-запрос — сделать?»

### 2.4. Cost-aware planning

**Время Dr. Nik — самое дорогое.** Минута переделки кода ≪ 10 минут переделки UI ≪ 1 час переделки внешних настроек ≪ день регистрации статуса самозанятого ≪ неделя ожидания модерации VK.

При выборе пути всегда явно показывать стоимость и риск **всех** вариантов, даже если один кажется очевидным. Decision — за Dr. Nik, не за Claude.

Если есть workaround на 5 минут (например, Kate Mobile token через vkhost.github.io) против правильного пути на 3 дня (бизнес-верификация) — **обязательно показывать оба** с trade-off.

### 2.5. Внешние зависимости — главный источник риска

API третьих лиц (VK, Kling, ProxyAPI) меняются чаще чем документация это отражает. Любое утверждение про их поведение требует свежей проверки.

Особенно опасны:
- Бизнес-верификации (невозвратные действия)
- Изменения типа учётной записи (физлицо → самозанятый → ИП)
- OAuth flows (VK перевёл с oauth.vk.com на id.vk.com в 2025, мы об этом узнали постфактум)
- Лимиты, квоты, scope (могут урезаться silently)

## 3. Антипаттерны (как НЕ делать)

### 3.1. ❌ «Phase 2 после YELLOW Phase 1»

**Как было в кейсе VK:** Phase 1 Discovery вернул YELLOW (5 положительных сигналов, 1 неподтверждённый — «scope wall/photos/video в реальности возможно урежется»). Claude согласился идти в Phase 2 implementation. После недели работы выяснилось что именно этот один YELLOW и есть блокер.

**Как надо:** YELLOW = stop. Перед Phase 2 — конкретные эксперименты которые закрывают unknowns. Каждый ≤30 минут. В VK-случае это был ручной OAuth в браузере на чужом тестовом приложении или curl-запрос с тестовым токеном.

### 3.2. ❌ «Архитектура без прототипа»

**Как было:** Code написал `core/vk_oauth.py` на ~200 строк со всей refresh-логикой, lock-механизмами, экспоненциальной задержкой, юнит-тестами **до того** как был получен хоть один реальный токен от VK.

**Как надо:** сначала одноразовый ≤30 строк скрипт который пробует получить токен. Получил с правильным scope → пишем production. Получил с урезанным scope → знаем за час вместо недели.

### 3.3. ❌ «Универсальные рекомендации из памяти»

**Как было:** Claude в начале посоветовал community-token для медиа в группе. Это было неверно — community-token не работает с `photos.getWallUploadServer` (VK error 27). Ошибка из памяти, не сверенная с актуальной документацией.

**Как надо:** перед советом по конкретному API-методу — web search актуального поведения **и** scope-требований. Особенно для VK, где требования к токенам менялись несколько раз.

### 3.4. ❌ «Доверие документации без сверки с практикой»

**Как было:** Claude доверял VK FAQ что «бизнес-верификация открывает расширенные scope». Не учёл что в FAQ имеется в виду статус ИП/самозанятого/юрлица, а профиль «физлицо» — это не бизнес.

**Как надо:** FAQ официальной документации = одна точка зрения. Свежие практические кейсы (Habr-статьи, GitHub Issues) — другая. Если они расходятся — практика важнее.

### 3.5. ❌ «Длинное ТЗ без декомпозиции»

**Как было:** ТЗ `CLAUDE_CODE_VK_OAUTH_001.md` — 428 строк, охватывал Discovery + Implementation + Server-deployment + Tests. Слишком много контекста для одной сессии Code.

**Как надо:** если ТЗ длиннее 300 строк — декомпозиция. Discovery — отдельное короткое ТЗ. Implementation — отдельное (только после GREEN). Server-deployment — отдельное (на будущее).

### 3.6. ❌ «Раннее тегирование»

**Как было:** ставили тег `v0.3.0-rc1` авансом, потом пришлось его перевешивать на pivot-коммит.

**Как надо:** тег ставится **только** после реальной end-to-end проверки в production-условиях. RC ставится только если есть зафиксированный план тестирования.

## 4. Регламент Discovery для внешних интеграций

Перед каждой задачей которая включает внешний API/сервис:

### 4.1. Структура `docs/<integration>-research.md`

```markdown
# Research: <название>
Дата: YYYY-MM-DD

## Источники
- URL + дата публикации + тип (официальная док / форум / GitHub Issue)

## Что подтверждено железно
- Факт 1 (источник)
- Факт 2 (источник)

## Что подтверждено косвенно
- Возможный факт 1 (намёк из источника, требует проверки)

## Unknowns
- Вопрос 1 (что именно неизвестно)
- Вопрос 2

## Главный вопрос
Прямая формулировка: «Работает ли X в нашем случае Y?»
Ответ: подтверждено / косвенно подтверждено / не подтверждено

## Вердикт
- 🟢 GREEN — все факты подтверждены, есть рабочий пример → реализация
- 🟡 YELLOW — есть 1-2 unknowns → СТОП, план закрытия unknowns
- 🔴 RED — фундаментальная блокировка → не реализуем, обсуждаем альтернативы

## (если YELLOW) План закрытия unknowns
1. Эксперимент 1 (≤30 минут): что сделать, что проверить, какой исход = GREEN
2. Эксперимент 2 …
```

### 4.2. Правило «дешёвых экспериментов»

Каждый эксперимент в плане закрытия unknowns должен:
- Занимать ≤30 минут реального времени Dr. Nik
- Не требовать невозвратных действий (регистраций, верификаций)
- Иметь явный бинарный исход — GREEN или конкретный fallback

Если эксперимент дороже — это не эксперимент, это уже часть реализации, и нужно понизить вердикт Discovery до RED или искать дешёвую альтернативу.

## 5. Регламент ТЗ для Claude Code

Каждое ТЗ обязательно содержит:

### 5.1. Структура

1. **Тип задачи** — discovery / implementation / fix / experiment / test
2. **Контекст** — что предшествовало, какие unknowns есть
3. **Цель** — что должно работать после
4. **НЕ-цели** — явный список того что НЕ делается
5. **Decision gates** — где Code останавливается и спрашивает Dr. Nik
6. **Шаги** — каждый имеет вход, выход, проверку
7. **Сценарии падения** — что делать если результат отличается от ожидаемого
8. **Финальный артефакт** — что появится в проекте

### 5.2. Ограничения длины

- ≤300 строк — норма
- 300–500 строк — звоночек, рассмотреть декомпозицию
- >500 строк — обязательная декомпозиция на 2+ ТЗ

### 5.3. Запрещённые формулировки в ТЗ

- «Реализуй всю инфраструктуру X» (без discovery)
- «Сделай красиво» (без критериев)
- «Если что-то пойдёт не так — придумай решение» (без альтернатив)
- «На своё усмотрение» (без явных рамок)

### 5.4. Discovery-фаза в ТЗ

Если ТЗ включает интеграцию с внешним API, **первая фаза всегда Discovery**, с **явным decision gate** до Phase 2.

Code НЕ может перейти в Phase 2 самостоятельно. Должен:
1. Закончить Discovery, выдать вердикт
2. Сообщить Dr. Nik
3. Дождаться явного «продолжай» от Dr. Nik (через Claude чата)

## 6. Регламент Product Pivot

Если по ходу реализации выясняется блокировка от внешнего сервиса:

### 6.1. Признаки что нужен pivot

- Документированное ограничение которое нельзя обойти кодом
- Стоимость обхода превышает ценность фичи (часы вместо минут)
- Все три альтернативных пути имеют серьёзные минусы

### 6.2. Что делать

1. **Стоп реализация немедленно** — не пытаться heroics-кодом обойти то что не работает
2. **Поднять флаг Dr. Nik** с:
   - Точной формулировкой что не работает (с error codes, документацией)
   - Списком альтернатив с trade-off каждой
   - Рекомендацией но не решением
3. **Дать выбор Dr. Nik:** продолжать пытаться / pivot / отложить
4. **Если pivot принят** — удалить мёртвый код, обновить docs, тег

### 6.3. Pivot — не поражение

Pivot — нормальная часть продуктовой разработки. **Цель проекта — публикация контента**, не «использование всех scope VK API». Если текст работает, а медиа нет — мы достигаем цели на 80%, и это уже success.

## 8. Кейс VK OAuth — что пошло не так (исторический)

### Хронология

| Этап | Что было | Что должно было быть |
|---|---|---|
| Старт | Claude посоветовал community-token для медиа | Web search документации VK 2025 → выяснилось бы что community-token не работает с photos.getWallUploadServer |
| Phase 1 Discovery | Вердикт YELLOW, идём в Phase 2 | YELLOW = stop. Сначала ручной OAuth-эксперимент в браузере (30 минут) |
| Phase 2 Implementation | Написан core/vk_oauth.py на 200+ строк, 20 тестов | Сначала 30-строчный одноразовый скрипт получения токена. Только при GREEN — production-код |
| OAuth flow | scope урезан до vkid.personal_info | Это можно было увидеть на первом же OAuth-запросе в браузере, до недели работы |
| Бизнес-верификация | Прошёл как «физлицо», без активации API | Проверка свежих кейсов на Habr показала бы что нужен ИП/самозанятый/юрлицо до подачи |
| Самозанятый профиль | Создан новый профиль, переподключение | Это всё могло быть избегнуто Discovery |
| Pivot | Текст-only, удаление media-функционала из публикации | Pivot правильный, но мог быть на 3 дня раньше |

### Стоимость ошибки

- **Время Dr. Nik:** 3 дня активной работы + перерывы
- **Время Claude (минут):** несколько сотен (на разговоры)
- **Время Code (часов):** 6-8 (на код который частично удалён)
- **Невозвратные действия:** регистрация бизнес-профиля VK с паспортными данными, создание трёх Standalone-приложений
- **Эмоциональная стоимость:** высокая (фрустрация, ощущение «опять не работает»)

### Уроки

1. **Не верить общим знаниям из памяти** про API третьих лиц. Всегда web search.
2. **YELLOW = stop, не GO.** Это правило написано большими буквами в этом файле.
3. **Дешёвый эксперимент до дорогой архитектуры.** 30 минут в браузере = 5 дней в коде.
4. **Свежие кейсы важнее официальной документации.** Habr-комментарии 2025–2026 показали бы блокировку.
5. **Невозвратные действия требуют явного согласия Dr. Nik с пониманием trade-off.** Не «давайте попробуем», а «вариант А стоит X с риском Y, вариант Б стоит Z, выбирайте».
